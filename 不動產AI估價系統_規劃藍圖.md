## 一、定位（白話）

### 1.1 系統長什麼樣

賞哥/夥伴**打開 `house-profile.html`** → 上傳謄本 → 解析資料 → A4 銷售資料表完成 → 多了一顆 **「🤖 生成估價建議書」** 按鈕 → 點下去 → AI 跑 1-3 分鐘 → 產出專業估價建議書（11 頁 HTML）→ 線上預覽可編輯 → 列印 PDF / 存進 Drive。

### 1.2 法律定位（重要）

- ✅ 文件標題寫「**市場行情參考**」，不寫「估價報告」
- ✅ 末頁附**完整免責聲明**
- ✅ 註明**非估價師法定報告**（依《不動產估價師法》）
- ✅ 提醒實際稅費以稅務機關為準
- ❌ 不能宣稱「估價」（避免違法）
- ❌ 不能給銀行做為貸款用途

### 1.3 個資簡化原則

收：物件地址（公開）、客戶尊稱（陳先生 / 王小姐）、物件詳細資料（公開）
不收：客戶全名、身分證、電話、生日、任何可識別個資

## 四、估價方法（從 v1.0 精選 + 簡化）

### 4.1 三大估價法

#### 📐 比較法（Sales Comparison）— 主力

**邏輯**：找相似案例，依差異調整推估。

**適用**：公寓 / 大樓 / 華廈、屋齡 < 30 年、交易熱絡區。

**資料來源**：內政部實價登錄 Open API（合法免費）。

**調整係數**：
- 屋齡差異：每差 1 年 ±1%
- 樓層差異：1 樓店面 +30~80%、頂樓加蓋 +10~20%、地下室 -40~60%
- 面寬深度（透天）：±5~15%
- 臨路條件：±10~30%
- 屋況新舊：±10%

**篩選**：1 km 內、近 12 個月、同類型、單價排除中位數 ±30%、時間衰減加權（半衰期 6 個月）。

#### 🏗️ 成本法（Cost）— 老屋透天用

**公式**：
```
土地價值 = 土地面積 × 市場地價
建物重建成本 = 建坪 × 當期營建單價
建物折舊 = 重建成本 × (屋齡 / 耐用年限) × 折舊率
建物現值 = 建物重建成本 - 建物折舊
估算總價 = 土地價值 + 建物現值
```

**適用**：屋齡 ≥ 25 年透天、工業廠房、特殊建物。

**耐用年限**：RC 50 年、SRC 60 年、加強磚造 35 年、木造 20 年。

**資料來源**：地政司公告地價 / 現值 Open API（既有 gisdawh 已可拿）+ 主計總處營建工程物價指數。

#### 💰 收益法（Income）— 店面 / 套房用

**公式**：
```
年淨收益 = (月租金 × 12) × (1 - 空置率) - 年營運費用
估算總價 = 年淨收益 / 資本化率（Cap Rate）
```

**台灣 Cap Rate 參考（2025）**：台北店面 2.0~2.5%、新北店面 2.8~3.5%、套房收租 2.5~3.5%、工業地產 4.0~5.5%。

**適用**：店面、辦公室、收租套房。

### 4.2 估價方法決策樹（自動選法）

```python
def select_methods(property):
    if property.type in ["大樓", "華廈", "公寓"]:
        if property.age < 30:
            return {"比較法": 1.0}
        else:
            return {"比較法": 0.7, "成本法": 0.3}
    elif property.type == "透天":
        if property.age >= 25:
            return {"比較法": 0.5, "成本法": 0.5}
        else:
            return {"比較法": 0.7, "成本法": 0.3}
    elif property.usage in ["店面", "辦公室"]:
        if property.has_rental_income:
            return {"比較法": 0.4, "收益法": 0.6}
        else:
            return {"比較法": 0.5, "收益法": 0.5}
    elif property.rental_type == "多套房":
        return {"收益法": 0.7, "比較法": 0.3}
```

### 4.3 結果調和

```
範例：屋齡 35 年透天
├─ 比較法: 2,580 萬 (權重 0.5)
├─ 成本法: 2,380 萬 (權重 0.3)
└─ 收益法: 2,450 萬 (權重 0.2)

加權平均: 2,494 萬
建議售價區間: 2,370 ~ 2,620 萬 (±5%)
```

## 七、房地合一稅試算（Phase 1 重點）

### 7.1 為什麼這是 Killer Feature

- 客戶最常問「**賣這個價格我實際能拿多少？**」
- 一般房仲很難當場算清楚（要查土增稅、房地合一稅、各項費用）
- AI 自動算 + 多情境比較 = 強差異化賣點

### 7.2 房地合一稅 2.0 計算

適用：2016/1/1 後取得的不動產

| 持有期間 | 個人境內稅率 | 個人境外稅率 |
|---|---|---|
| 2 年內 | 45% | 45% |
| 2-5 年 | 35% | 35% |
| 5-10 年 | 20% | - |
| 10 年以上 | 15% | - |

**自用住宅優惠**：稅率 10%、免稅額 400 萬，需符合：
- 本人 / 配偶 / 未成年子女設籍滿 6 年
- 持有期間未出租或營業
- 6 年內未曾使用過此優惠
- 出售前未有其他自用住宅

### 7.3 賣方完整成本清單


## 十一、不採用 v1.0 方案的理由（記錄起來，未來別重討論）

### 11.1 為什麼不上 NAS

- 跟「資安防護筆記」原則衝突（家用網路五關陷阱、QNAP 對外開 port = 勒索病毒攻擊面）
- Drive + Worker 已經夠用（喬喬實戰驗證通過）
- 多一層維護成本（NAS 斷電 / 系統更新 / Docker 容器爆炸）
- 沒明顯收益

### 11.2 為什麼不用 Streamlit + Python

- 賞哥技術棧是 HTML/JS（已熟）
- Streamlit 美觀度有限，Python 後端要學新框架
- 既有 `house-profile.html` 已經有完整 A4 + 列印 + Drive，重做沒道理

### 11.3 為什麼不用 PostgreSQL

- 個人工具規模 Drive JSON 完全夠
- 多一層資料庫 = 多一層備份 / 遷移 / 維護成本
- 未來真要上 PostgreSQL（多人版）再說

### 11.4 為什麼不用 Claude API

- Gemini 3.1 Pro 推理力 ≈ Claude Opus 4.7
- Claude 貴 2 倍以上
- 既有架構 shang-gemini-proxy 已建，切 Claude 要重做
- 沒明顯收益

### 11.5 為什麼不用 OpenAI 圖片

- CSS 設計 + Static Maps 已夠美
- AI 出圖品質不穩（手指 6 根、文字亂碼）
- 多一支付費 + Worker
